

### 1. HTTP 4xx Errors (GAIL Microservices)
#### Initial Response:
1. Acknowledge the alert in the monitoring system.
2. Notify the on-call team and stakeholders.

#### Investigation Steps:
1. Access AWS:  
   * Log in to the AWS Dashboard: [AWS Console](https://dwp-sso.awsapps.com/start/#/?tab=accounts)
2. Access Grafana Dashboard:  
   * [Grafana](https://grafana.tooling.pdu-dev.hcs-eks.dwpcloud.uk/d/AII-GAIL/aii-gail?orgId=1&refresh=30s)
3. Review recent deployments, pipelines, or changes to the application.
4. Verify the status of dependent services and APIs.
5. Check logs in CloudWatch:  
   * [CloudWatch Logs](https://eu-west-2.console.aws.amazon.com/cloudwatch/home?region=eu-west-2#logsV2:logs-insights)
6. Investigate if API gateway misconfigurations are causing the errors.

#### Resolution Steps:
1. Identify the source of 4xx errors in the logs.
2. Verify API gateway routing and authentication configurations.
3. Check for misconfigured security rules blocking API access.
4. Roll back recent deployments if necessary.
5. Fix application logic causing invalid requests.
6. Ensure dependent services are operational.

### 4. Health Check - One AZ Down (Application Load Balancer - Bedrock Account)
#### Initial Response:
1. Acknowledge the alert in the monitoring system.
2. Notify the on-call team and stakeholders.

#### Investigation Steps:
1. Verify AWS ALB logs for health check failures.
2. Check target group status in the AWS console.
3. Ensure instances in the affected AZ are running and healthy.
4. Check for networking or security group misconfigurations.

#### Resolution Steps:
1. Restart any failing instances in the affected AZ.
2. Redistribute traffic across available AZs.
3. Verify and update auto-scaling policies if necessary.
4. Ensure new instances are registered correctly with the ALB.
